# WYnightDemo开发日志
## 2026.4.1

> **遇到问题**

1. 从互联网上获取的动作包模型无法正确渲染
2. 动画拥有非常多角度的动作，如何混合
3. 从普通跑到疾跑，怎么才能做到既躲避攻击又进行顺滑的疾跑
4. 插入疾跑后，发现角色一直疾跑

> **解决方法**

1. 渲染设置为URP，即Unity自带内置渲染，然后拖动材质球到对应模型即可
2. 只选取跑步前进、疾跑前进、待机，使用混合树进行混合，视角、体感上均较为流畅
3. 插入一个闪避，类似于大多数游戏的冲刺（闪避）后才开始疾跑
4. 不仅检测输入shift或鼠标右键，同时还要检测是否为跑步状态，使疾跑状态无法一直闪避

> **我的思考**
1. 模型不能拿来即用，还需要进行基本的配置，要酌情考虑素材的使用
2. 混合树能让首尾帧相近的动作尽量过渡流畅，可以利用这个拼接一些动画，比如连招
3. 这想法来源于大部分游戏的做法，要记得以前玩过的各类3A游戏的3C
4. 记得限制闪避条件

## 2026.4.2

> **遇到问题**

1. 闪避只能触发一次，无法连续闪避
2. 连续闪避时，动画从疾跑姿态突然切回站立姿态，衔接不流畅
3. 第二次闪避后回到移动状态时会出现跳帧
4. 闪避后没有预输入机制，手感差

> **解决方法**

1. 增加闪避次数计数器，设计为连续两次闪避后才进入2秒冷却，冷却结束后重置次数
2. 使用强制播放动画并从第13帧开始，跳过站立的起手部分，直接从闪避动作中间衔接
3. 闪避动画在isDodge在设置为false之前，就已经播放完了，然后又因为动画为设置为循环，所以播放了闪避动画了一小段，然后又回到移动混合树导致动画抽帧
4. 添加预输入变量，在闪避动画中记录玩家按键，动画结束后自动触发下一次闪避

> **我的思考**

1. 两次闪避给了玩家容错空间，1.5s冷却防止滥用，这是一种资源管理设计，让玩家需要判断何时用掉两次闪避何时保留，增加了战斗的策略深度
2. 连续闪避时从疾跑跳回站立的不流畅，本质是动画起手姿态没有匹配当前角色状态。跳过站立起手部分直接从闪避动作中间衔接，说明动画剪辑点可以根据实际需求灵活调整
3. 闪避动画结尾和疾跑循环的跳帧问题，要注意动画是否被提前切换了，可以通过把动画循环关闭，如果卡住了就是动画被提前结束了
4. 预输入是动作游戏的基本功，让玩家的操作被记住而不是因为动画播放中就丢失输入。很多游戏手感差就是因为预输入窗口太短或根本没有

## 2626.4.5

> **遇到问题**
1. 快速点击攻击键时，角色无法按顺序打出1-2-3-4连段，经常跳过中间段或连段中断。
2. 攻击动画期间角色可以平移和旋转，破坏了攻击动作的严肃性和打击感。
3. 移动输入每帧触发 `turnToMove`，导致动画状态机被持续干扰，攻击输入无法正常响应。
4. 动画事件无法正确绑定，报错 `has no receiver`。

> **解决方法**
1. 使用整型计数器记录缓冲攻击次数，在 `Update` 中每帧检查，当角色处于可攻击状态时依次执行，配合 `ResetTrigger` 防止 Trigger 累积。
2. 通过 `IMovementProvider` 接口添加 `CanMove` 和 `CanRotate` 属性，攻击时禁用，收刀帧恢复。在 `PlayerRotate()` 中同时检查 `CanMove`，确保攻击期间完全禁止转向。
3. 检测移动状态变化，仅在从静止到移动的瞬间触发一次 `turnToMove`，而非每帧触发，即额外在DisableMoveAndRotate方法加一个重置trigger
4. 动画绑定的事件上的方法名忘记改了，改回去就好

> **我的思考**
1. 预输入系统用计数器比布尔值更可靠，快速点击时多次输入需要被完整记录而非合并，这是动作游戏手感的基础。
2. 之前优化的代码结构现在体现了作用，调用接口让各组件联动更加方便同时又能降低耦合
3. 通过监视动画状态机能够精准定位问题，反复触发一定是忘了重置
4. 别乱动方法名，特别是绑定了动画事件的方法

> **遗留问题**
1. 角色攻击无法移动是实现了，但也无法旋转了，根本无法在攻击连段中转向，因为在快速左键下连段没有间隔，需要尝试制作旋转预输入

## 2626.4.7

> **遇到问题**
1. 攻击期间完全无法转向，快速连点时两段攻击之间没有转向窗口。
2. `GetInput` 事件只在按键按下瞬间触发，按住方向键无法持续刷新旋转缓冲。
3. 攻击动画较长时，旋转缓冲在攻击结束前超时被清除，导致转向失败。
4. 两次攻击间隙太短，转向只执行 1-2 帧就被下一次攻击的 `DisableMoveAndRotate` 打断。
5. 需要实现"每次攻击最多转向一次"的限制，防止攻击中反复转向。
6. 使用 `transform.rotation = targetRotation` 瞬间转向，视觉效果太生硬。

> **解决方法**
1. 在 `Update` 中每帧检查 `playerInput.magnitude`，不依赖事件触发，持续刷新缓冲方向和计时器。
2. `StartRotateBuffer` 移除 `if (rotateBuffered) return` 条件，允许按住方向键时每帧刷新缓冲。
3. `CheckRotateBufferTimeout` 添加 `if (!CanRotate) return`，攻击期间不清除缓冲。
4. `ContinueRotateBuffer` 移除 `if (!CanRotate) return`，让转向在多段攻击间隙中持续执行。
5. 新增 `rotatedThisAttack` 布尔变量：`EnableAttackRotate()` 重置为 `false`，转向后立即标记 `true`，本次攻击不再转向。
6. 使用 `Quaternion.Slerp` 配合倍速参数实现快速平滑转向：
   ```csharp
   float fastRotateSpeed = rotateSpeed * attackRotateSpeedMultiplier;
   transform.rotation = Quaternion.Slerp(transform.rotation, targetRotation, fastRotateSpeed * Time.deltaTime);
   ```

> **我的思考**
1. 事件驱动适合"按下瞬间"的响应（如攻击），轮询检测适合"持续状态"的响应（如移动、转向缓冲）。这次将旋转缓冲从事件驱动改为 Update 轮询，解决了按住方向键无法持续刷新缓冲的问题。
2. 之前为了"攻击期间禁止移动和旋转"，在入口处直接 `return`。但旋转预输入恰恰需要在攻击期间执行。解决方法是区分"禁止手动旋转"和"允许预输入旋转"，通过 `rotatedThisAttack` 标记实现"攻击中只转一次"的限制。
3. 超时检查只在"可旋转"状态下执行，攻击期间不清除缓冲。这确保了长动画结束后缓冲依然有效，同时避免了非攻击状态下缓冲永久残留。
4. 瞬间转向虽然功能正确，但缺乏重量感。使用 `Slerp` 配合倍速参数，既保持了响应速度，又保留了动作游戏的流畅感。

## 2026.4.9

> **遇到问题**
1. 按住方向键攻击结束后，角色会立刻切换到移动动画，打断连招。需求是：只有纯粹想移动（无攻击输入、无预输入）时才切换到移动，否则只转向不移动。
2. 攻击动画最后一帧触发收刀事件，导致攻击结束立刻收刀，没有持刀停留的节奏感。
3. 攻击结束姿势（左脚在前）与待机姿势（右脚在前）不一致，直接过渡会导致脚步瞬间切换，不自然。
4. Any State 到镜像待机的 1.5 秒过渡期间，无法用 Trigger 打断，导致玩家操作无响应。

> **解决方法**
1. 在 `TurnToMove()` 的判断条件中加入 `!isAttacking && !attackBuffered`，确保只有在无攻击意图时才触发移动切换。
2. 创建"持刀待机"过渡状态，攻击结束后先进入该状态停留 0.3 秒，再触发收刀。
3. 利用 Animator 的 Mirror（镜像）功能，将待机动画镜像播放，左脚在前变为右脚在前，无需修改原始动画资源。
4. 创建"过渡中转状态"（Transition_Buffer）作为跳板：Any State 瞬间进入该状态，再从该状态出发 1.5 秒混合到镜像待机。攻击 Trigger 可以从该状态打断长混合，实现"视觉平滑 + 逻辑可打断"。

> **我的思考**
1. 过渡中转状态的设计让我理解了 Animator 的工作原理——过渡本身是"无实体"的，无法作为打断的起点。通过引入一个逻辑上的"跳板状态"，可以在保留视觉长混合的同时，让状态机有明确的"当前状态"来响应打断请求。这
2. Mirror 镜像功能是处理左右对称动画的高效工具。与其复制修改动画文件，不如在运行时动态控制镜像参数。这不仅节省了资源，也便于后续维护——如果待机动画本身被替换，镜像版本自动同步更新。
3. 攻击后的短暂停留为玩家提供了"确认攻击结束"的视觉反馈，也为连招输入留出了缓冲窗口。这种"微停顿"是动作游戏手感的隐性组成部分。
